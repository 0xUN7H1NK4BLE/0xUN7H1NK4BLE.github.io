Process vs Thread

FUNDAMENTALS

Process

A program in execution. Has its own PID, memory space, handles, and security context.

Examples: notepad.exe, chrome.exe, sshd

Every process has at least one thread.

Thread

The smallest unit of execution within a process. Multiple threads share the same memory space.

All threads in a process share the same memory and resources.

Attackers inject threads into other processes (process injection).

SOC Insight: Always check parent-child relationships. winword.exe → powershell.exe is suspicious.

Sysmon: Pros & Cons

FUNDAMENTALS

Advantages

Free and supported by Microsoft — no license cost, production-ready, included with Sysinternals.
Minimal performance impact — runs as a kernel driver, typical CPU usage is <1%. Can be deployed on thousands of endpoints.
Granular visibility — command lines, network connections per process, DNS queries, file hashes, registry changes. All in one log source.
Configurable filtering — XML config to include/exclude events. Reduce noise by filtering known-good behavior (e.g., Windows Update, antivirus scans).
Integrates everywhere — events go to the standard Windows Event Log, consumed by any SIEM (Splunk, Elastic Sentinel, Wazuh, etc.).

Disadvantages

Driver conflicts — SysmonDrv.sys can conflict with some antivirus/firewall drivers. Requires testing before deployment.
Log volume explosion without a good config — default config logs everything. Expect 5-10 GB/day per host with the default config. A curated config is essential.
No alerting on its own — Sysmon only generates logs. It does not alert, correlate, or notify. You need a SIEM or EDR to analyze the data.
Can be unloaded by kernel-mode malware — a sufficiently privileged attacker can stop or tamper with the Sysmon driver. It's not a security boundary.
No detections — only telemetry — Sysmon provides data; your analysts and rules provide detection. It's only as good as your ability to query and analyze.

Verdict: Deploy Sysmon everywhere, but pair it with a good config (SwiftOnSecurity) and a SIEM to consume the logs.

Sysmon Config: How It Works

DEEP DIVE

Sysmon uses an XML config file that tells it what to log and what to ignore. Understanding the config = understanding what you're (not) seeing.

<!-- SwiftOnSecurity config — sample rule -->
<Sysmon schemaversion="4.90">
<EventFiltering>

<!-- Log all process creation EXCEPT known-good -->
<ProcessCreate onmatch="exclude">
<CommandLine condition="is">C:\Windows\system32\svchost.exe -k -p</CommandLine>
<Image condition="end with">\Windows\System32\taskhostw.exe</Image>
</ProcessCreate>

<!-- Log all network connections EXCEPT known-good -->
<NetworkConnect onmatch="exclude">
<Image condition="begin with">C:\Windows\System32\svchost.exe</Image>
<DestinationIp condition="is">127.0.0.1</DestinationIp>
</NetworkConnect>

<!-- Log all DNS queries EXCEPT known-good -->
<DnsQuery onmatch="exclude">
<Image condition="begin with">C:\Windows\System32\sppsvc.exe</Image>
</DnsQuery>
</EventFiltering>
</Sysmon>

`onmatch="exclude"` means: log everything except these. This is the default. You add exclusions to reduce noise.

Common conditions: is (exact match), begin with, end with, contains, is any (list). Build exclusions carefully — too broad = you miss attacks.

Processes & svchost.exe

CORE

Every Windows process has an EPROCESS (kernel tracking block), a PEB (user-mode structure with DLL list, command line), and a TEB per thread. Attackers manipulate these to hide code.

The #1 process to know: svchost.exe (Service Host)

Check these four fields in Sysmon EID 1 or Process Explorer. If any doesn't match the legit pattern, treat it as malicious until proven otherwise.

Using Task Manager

BASIC

Task Manager gives you a quick view of running processes. Press Ctrl+Shift+Esc to open it. It's the first tool you reach for during triage.

What to check when investigating:

  • Processes tab — look for unfamiliar process names, high CPU/memory usage, or processes running under unusual usernames (e.g., svchost.exe running as .\victim instead of LOCAL SERVICE)
  • Startup tab — see what runs at boot. Disable anything suspicious
  • Details tab — right-click columns → add Command Line and Parent Process. This turns Task Manager into a mini-Process Explorer. Check if svchost.exe has -k flag and if parent is services.exe
  • Right-click any process → Search online to quickly check if it's known malware

Limitation: Task Manager doesn't show loaded DLLs or handles. When you need that detail, switch to Process Explorer.

LSASS: The Crown Jewel

ATTACK

LSASS handles all authentication — every password hash, Kerberos ticket, and plaintext credential passes through its memory. This makes it the #1 target after initial compromise.

Mimikatz
sekurlsa::logonpasswords

ProcDump
procdump.exe -ma lsass.exe

DCSync
lsadump::dcsync

Detection in Sysmon: EID 1 where Image contains procdump and CommandLine contains lsass.exe. Also EID 11 (FileCreate) for *.dmp files. If you see this — isolate the host immediately, every credential on that machine is compromised.

Modern Windows runs LSASS as PPL (Protected Process Light), which blocks direct memory reads. Attackers bypass it with kernel drivers or by running as SYSTEM.

DLL Search Order Hijacking

ATTACK

When an app loads a DLL, Windows searches in order: app's directory → System32 → System → CWD → PATH. If a DLL isn't in System32, the app's own directory is checked first. Drop a malicious DLL there with the right name → the app loads your code.

# Legit app loading a missing DLL
App dir: C:\Program Files\App\app.exe
looks for helper.dll → NOT FOUND
System32: helper.dll → NOT FOUND
App dir: helper.dll → FOUND! (attacker placed it)
# Detection in Sysmon EID 7
ImageLoaded: C:\Program Files\App\helper.dll
Image: C:\Program Files\App\app.exe

Check: Is this DLL signed?
Does it belong in this path?

Query Sysmon EID 7 for DLLs loaded from non-C:\Windows\ paths. Cross-reference with a baseline of known-good DLL locations.

Process Injection Techniques

ATTACK

Running malicious code inside a legitimate process to evade detection. Three main types:

DLL Injection

CreateRemoteThread + LoadLibrary — injects a DLL into the target process.
Detection: Sysmon EID 8 (CreateRemoteThread).

APC Injection

Queues code to an existing thread. No new thread = stealthier.
Detection: ETW Threat Intelligence EID 12. Hard to detect with Sysmon alone.

Process Hollowing

Spawns a legit process (suspended), unmaps its code, replaces with malware.
Detection: Check memory regions in Process Explorer — hollowed processes show no loaded image.

Detecting DLL Injection

DETECTION

Sysmon EID 8 is your strongest signal. Here's what it looks like in practice:

# Sysmon EID 8 — remote thread created
SourcePID: 4520 (malware.exe)
TargetPID: 3488 (explorer.exe)
StartAddr: 0x7FFE...
StartModule: kernel32.dll

Source != Target = injection
# Sysmon EID 1 — the injector's origin
Image: C:\Users\victim\AppData\Local\Temp\mal.exe
ParentImage: OUTLOOK.EXE

Temp dir + EID 8 = confirmed

EID 8 + EID 1 from Temp/Local/AppData = process injection. Isolate the host and preserve memory for forensic analysis.

Process Hollowing

DETECTION

The attacker spawns a legitimate process suspended, unmaps its code, writes malicious code, and resumes it. The process name looks completely legit — but its memory is attacker-controlled.

# How hollowing works (conceptual)
1. CreateProcess("notepad.exe", CREATE_SUSPENDED) → legit notepad.exe, PID 4520
2. NtUnmapViewOfSection(hProcess, baseAddr) ⟵ rip out notepad's code
3. VirtualAllocEx + WriteProcessMemory(hProcess) ⟵ write malware in its place
4. SetThreadContext + ResumeThread(hProcess) ⟵ "notepad.exe" runs attacker code

How to detect it: In Process Explorer, right-click the process → Properties → Image tab. If the process says "Unable to load image" or shows no loaded modules, it's likely hollowed. Also check Sysmon EID 1 — a hollowed process will have an unusual parent (e.g., notepad.exe spawned by cmd.exe).

Business Email Compromise

ATTACK

BEC is the most financially damaging cybercrime — no malware, no links, just social engineering. Attackers impersonate executives or vendors to trick employees into wiring money or sharing credentials.

CEO Fraud

Email from "CEO" to finance: "I need an urgent wire transfer for the acquisition. Keep this confidential."

Vendor Compromise

Attacker compromises a vendor's email → sends invoice with changed bank details → payment goes to attacker.

Account Takeover

Employee's email credentials stolen → attacker monitors inbox → inserts fake requests at the right moment.

Gift Card Scam

"CEO" asks an employee to buy gift cards for a client gift and send photos of the codes. No malware involved.

Detection: Look for emails with Reply-To or Return-Path different from the From domain. Check email authentication (SPF, DKIM, DMARC failure). Monitor for unusual login locations.

Ransomware

ATTACK

Ransomware is the most visible cyber threat. Understanding how it operates is critical for detection and response.

# Typical ransomware timeline (from SOC data)
T+0:00 Phishing email opened by user
T+0:05 Macro-enabled Office doc → downloads Cobalt Strike beacon
T+0:15 Beacon calls out to C2 (usually over HTTPS to a legitimate-looking domain)
T+1:00 Attacker enumerates domain: whoami, net group "Domain Admins" /domain
T+2:00 Lateral movement via PsExec or WMI — spreads to file servers
T+2:30 Deletes Volume Shadow Copies: vssadmin delete shadows /all
T+2:35 Deploys ransomware via Group Policy or scheduled task
T+2:36 Encryption begins — files renamed with .lockbit extension
T+2:40 Ransom note dropped everywhere: !README!.txt

Detection opportunities:

  • vssadmin delete shadows — almost never legit, alert on it
  • PsExec/WMI from a workstation to a server — lateral movement indicator
  • Mass file renames / new file extensions — EDR detects this
  • C2 beaconing (Sysmon EID 3 + 22) — early detection before encryption

Supply Chain Attacks

ATTACK

Attackers compromise a trusted vendor or tool — and everyone who uses it gets infected. You don't have to be the direct target; you just have to use the compromised product.

SolarWinds (2020)

Attackers inserted malware into SolarWinds Orion updates. 18,000 organizations downloaded the backdoored update. Impact: FireEye, US government agencies breached.

Log4j (2021)

Vulnerability in a widely used Java logging library. Hundreds of millions of applications exposed. Impact: Remote code execution in everything from Minecraft to enterprise apps.

Codecov (2021)

Attacker modified Codecov's Docker image — every customer who ran CI/CD pipelines had environment variables (including cloud keys) exfiltrated.

3CX (2023)

Attackers compromised the 3CX build pipeline — signed software update delivered malware to 600,000 customers. Supply chain meets trusted signing.

Detection is difficult because the software is legitimately signed. Mitigation: software inventory, behavioral detection, restrict what vendor processes can do.

Snort: Three Modes

FUNDAMENTALS

Snort runs in three distinct modes. Most SOC analysts will work with NIDS mode alerts, but knowing all three helps in lab testing.

Sniffer

Reads packets from the wire and prints them to stdout. Useful for debugging, not production.

Packet Logger

Logs packets to disk in tcpdump format. Can replay later for analysis.

NIDS

Evaluates traffic against rules and generates alerts. This is the mode you'll use in a SOC.

# Sniffer: read packets live
$ sudo snort -v -i eth0

# Packet Logger: write to disk
$ sudo snort -l /var/log/snort -i eth0

# NIDS: load rules and alert
$ sudo snort -c /etc/snort/snort.conf -i eth0 -A fast

Snort Rule Syntax

RULES

Every Snort rule has a rule header (action, protocol, source, destination) and rule options (what to match inside the packet).

# Rule structure
alert tcp $HOME_NET any -> $EXTERNAL_NET $HTTP_PORTS (
msg:"ET MALWARE Cobalt Strike Beacon Detected";
content:"|0d 0a|Server: Apache|0d 0a|"; distance:0;
sid:2025001; rev:1;
classtype:trojan-activity;
reference:url,threatfox.example.com;
)
FieldPurposeExample
alertAction: alert, log, drop, rejectAlert the SOC
tcpProtocol: tcp, udp, icmp, ipTCP-based attack
$HOME_NETSource/destination variableYour internal network
contentMatch byte sequence in payloadHTTP header pattern
sidUnique rule IDUse 1,000,000+ for custom rules
classtypeAttack classificationMaps to priority

The Emerging Threats (ET) open rule set ships thousands of rules. You can write custom rules to detect internal threats your environment cares about.

Reading Snort Alerts

ALERTS

Snort writes alerts to /var/log/snort/alert (fast mode) or alert.json (unified2). Every alert includes the matching rule, source/destination, and timestamp.

# /var/log/snort/alert — fast mode output
07/06-10:15:23.123456 [**] [1:2025001:1] ET MALWARE Cobalt Strike Beacon Detected [**]
[Classification: Trojan Activity] [Priority: 1]
TCP 10.0.0.5:49152 -> 185.234.72.18:443

07/06-10:15:24.654321 [**] [1:2025002:1] ET POLICY Outbound DNS Query to Suspicious Domain [**]
[Classification: Policy Violation] [Priority: 3]
UDP 10.0.0.5:54321 -> 192.168.1.2:53

Alert fields decoded:

  • [1:2025001:1] — generator_id:sid:revision. The SID identifies the exact rule.
  • Priority 1 — Most severe (attempted admin, trojan). Priority 3 is informational.
  • Source → Destination — IP and port of both sides of the connection.

Correlate Snort alerts with Sysmon EID 3 (network connection) in your SIEM. If both fire for the same IP, confidence is high.

MATH

True Positive (TP)

Correct alert on malicious behavior

False Positive (FP)

Alert on benign behavior

True Negative (TN)

No alert on benign behavior

False Negative (FN)

No alert on malicious behavior (worst case)

Precision = TP / (TP + FP) -- "How many alerts are real?"

Recall = TP / (TP + FN) -- "How many attacks do we catch?"

F1 = 2 x (Precision x Recall) / (Precision + Recall) -- "Balanced score"

SOC analysts care most about FP reduction. A noisy rule gets ignored or disabled.

Tuning Detections

OPTIMIZE

Every detection starts noisy. Tuning reduces FP without losing TP.

Whitelisting

Exclude known-good paths, users, processes. Skip legitimate services.

Thresholding

Alert only after N events in X minutes. Reduces noise from one-off actions.

Context Filtering

Add parent process validation to separate phish from admin execution.

Layered Detection

Detect same technique at different levels. If one layer fires alone, lower severity.

Detection as Code

DaC

Treat detection rules like software: version control, code review, CI/CD pipeline.

Git Repository

Store all Sigma/YARA rules in a Git repo. Track changes, authors, rollbacks.

Code Review

Peer review every rule before deploy. Catch logic errors and FP gaps.

CI/CD Pipeline

Auto-test, convert, deploy. GitHub Actions validate syntax, run Atomic tests.

DaC prevents "who wrote that rule?" and "why was this changed?" mysteries.

What is EDR?

CORE

Endpoint Detection & Response (EDR) continuously monitors endpoints for suspicious behavior and enables rapid investigation and response.

Detection

Process Trees

INVESTIGATE

The process tree is the most powerful EDR investigation tool. It shows every process spawn in parent-child relationship.

explorer.exe (PID 4520) └─ winword.exe (PID 6124) // User opened phishing doc └─ cmd.exe (PID 6780) // Macro launched cmd └─ powershell.exe (PID 6902) // Suspicious └─ rundll32.exe (PID 7001) // Beacon loaded └─ ${APPDATA}\updater.exe (PID 7100) // Malware

Each level adds context. WinWord → cmd → PowerShell is never legitimate in a SOC environment.

Key EDR features: collapse/expand subtrees, search within tree, export as JSON.

Detecting EDR Bypass Attempts

DETECT

Most bypass attempts leave telemetry artifacts. Train your eyes to spot them.

Tamper Alerts

EDR agent stopped, uninstalled, or driver disabled. Always escalate. Tamper = attacker is actively hiding.

Unsigned Drivers

Kernel driver load with invalid signature. EDR bypass tools often load malicious drivers (e.g., ProcExp bypass).

ETW Disable Events

Event ID in Sysmon/ETW indicating provider was disabled. Often done right before credential dumping.

Suspicious Call Stack

Process with no loaded DLLs or unusual call stack (no ntdll/kernel32). Indicates direct syscall execution.

What is Malware Analysis?

CORE

Malware analysis is the process of examining a suspicious file or URL to determine what it does, how it works, and how to detect it.

Static Analysis

Examining the file without executing it. Strings, PE headers, hashes, metadata. Safe, fast, no sandbox needed.

Dynamic Analysis

Running the file in an isolated sandbox. Observe behavior: processes, network, file system, registry.

Memory Analysis

Dumping and analyzing process memory. Find injected code, hidden processes, encryption keys.

SOC analysts mainly do static+triage + online sandbox analysis. Deep reverse engineering is for the malware analysis team.
