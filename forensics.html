The Nepal Landscape

Real challenges that can make investigations slower or harder.

05 / Anti-Forensics

Anti-Forensic Techniques (1/6)

Methods criminals may use to hide, change, or destroy evidence — and how investigators respond.

What “anti-forensics” means

  • Hide evidence (make it hard to find).
  • Change evidence (make it inaccurate or misleading).
  • Destroy evidence (make it unrecoverable).
  • Confuse the timeline (mislead investigators).

Investigator mindset

  • Assume important data might be missing on purpose.
  • Validate everything with multiple sources (files, logs, backups).
  • Preserve evidence first: image, hash, and document.
  • Explain findings clearly: what you know, what you infer, and what you cannot prove.
05 / Anti-Forensics

Data Wiping & Secure Deletion (2/6)

Deleting a file is not the same as wiping it.

Common tactics

  • Normal delete: removes the reference, data may remain on disk.
  • Wipe tools: overwrite blocks to reduce recovery chances.
  • Factory reset: can remove user data, but artifacts may remain (depends on device/OS).
  • Disk encryption + key destruction: makes data unreadable without the key.

Investigator response

  • Look for traces: file system logs, USN journal, shadow copies, app caches.
  • Check external sources: cloud sync, email, messaging backups.
  • Correlate time: when was wiping tool installed/run?
  • Document limits: some wiping/encryption can make recovery impossible.
05 / Anti-Forensics

Log Tampering & Timeline Tricks (3/6)

Attackers may try to erase or rewrite history.

Examples

  • Clearing system logs (Windows Event Logs, Linux auth logs).
  • Changing timestamps (“timestomping”).
  • Disabling logging / deleting browser history.
  • Changing timezone or system clock to confuse order of events.

How to counter

  • Cross-check timestamps: file system + app logs + network logs.
  • Look for “gaps” or “resets” in log sequences.
  • Compare device time vs. server time (email, cloud, VPN, routers).
  • Use multiple artifacts: prefetch, registry, browser cache, database records.
05 / Anti-Forensics

Encryption & Obfuscation (4/6)

Strong encryption can protect privacy — and also prevent lawful access without keys.

What you might see

  • Encrypted disks/containers (BitLocker, VeraCrypt, FileVault).
  • Password-protected archives (zip/rar) and “hidden volumes”.
  • Encoded/packed data (base64, custom formats).
  • Encrypted messaging apps with disappearing messages.

Investigation approach

  • Collect keys where lawful: memory capture, keychain artifacts, TPM state.
  • Look for unencrypted copies: thumbnails, exports, cloud sync.
  • Use context: file names, sizes, access times, mount history.
  • Report clearly when data cannot be decrypted.
05 / Anti-Forensics

Hiding Data (5/6)

Evidence can be hidden inside normal-looking files or locations.

Common hiding places

  • Steganography (data hidden inside images/audio).
  • Hidden partitions or “unallocated space”.
  • Alternate data streams (NTFS) / unusual filesystem features.
  • “Looks harmless” files (renamed extensions, fake headers).

How to detect

  • Check signatures (magic bytes) vs. file extensions.
  • Measure anomalies: file size, entropy, unusual metadata.
  • Search for known patterns and artifacts (thumbnails, caches).
  • Use carving and timeline correlation.
05 / Anti-Forensics

Living-Off-The-Land & Fileless Activity (6/6)

Some attacks avoid obvious malware files and run in memory or using built-in tools.

What it looks like

  • Scripts executed from memory (PowerShell, WMI, macros).
  • Abuse of legitimate admin tools (remote management).
  • Short-lived processes that disappear quickly.

Evidence sources

  • Memory images, process lists, command history.
  • Event logs, scheduled tasks, registry run keys.
  • Network telemetry: DNS, proxy, firewall logs.
06 / Digital Evidence

Digital Evidence: Overview (1/6)

Digital evidence is any information stored or transmitted in digital form that can support an investigation.

Two big categories

  • Volatile: disappears when power is off (RAM, running processes).
  • Non-volatile: stays stored (disk, phone storage, cloud files).

What makes it “good” evidence?

  • Collected legally and ethically.
  • Integrity preserved (hash + chain of custody).
  • Repeatable process (another expert can verify).
  • Clear explanation (technical → understandable).
06 / Digital Evidence

Device Evidence (2/6)

Computers and storage media contain system history, files, and hidden artifacts.

Examples

Documents, photos, downloads
Browser history, cache, cookies
System logs, registry, plists
USB connection history

Best practice

  • Image the drive (don’t “browse” the original).
  • Hash the image and keep notes.
  • Analyze on copies.
06 / Digital Evidence

Network Evidence (3/6)

Traffic and logs can show communication, intrusions, and data exfiltration.

Sources

  • PCAP packet captures
  • Firewall / router / VPN logs
  • DNS logs and proxy logs
  • Server access logs

What you can learn

  • Who talked to who (IP/ports)
  • When (timestamps)
  • How (protocols)
  • Sometimes what (payloads — depends on encryption)
06 / Digital Evidence

Mobile & App Evidence (4/6)

Phones store call logs, messages, locations, and app databases.

Typical artifacts

  • SMS, chats (often stored as SQLite databases)
  • Call history and contacts
  • App caches (images, voice notes, files)
  • Location history (if enabled)

Important note

Mobile evidence is sensitive. Always follow legal authority and privacy rules for the case.

06 / Digital Evidence

Cloud Evidence (5/6)

Data stored on services (email, drives, social media) often includes logs and history.

Examples

  • Email accounts (headers, access history)
  • Cloud storage (Drive/Dropbox/OneDrive)
  • Social media messages and posts
  • Audit logs (admin consoles)

Challenges

  • Jurisdiction and legal process
  • Short log retention
  • Encryption and account recovery
  • Provider cooperation
06 / Digital Evidence

Media, IoT & Other Evidence (6/6)

Photos, videos, smart devices, and sensors can contain important context.

Examples

EXIF metadata (camera, GPS)

07 / Procedure

Investigation Procedure: Preparation (1/9)

Before touching evidence: confirm authority, scope, tools, and documentation plan.

Checklist

  • Legal authority and scope (what you can collect)
  • Case ID, labels, chain-of-custody forms
  • Write blockers, storage, power supplies
  • Plan: volatile-first vs powered-off collection

Goal

Make your work repeatable, legal, and easy to explain later.

07 / Procedure

Identification (2/9)

Find what sources of evidence exist and decide what matters for the case.

Evidence sources

  • Devices: laptops, phones, USBs, DVRs
  • Accounts: email, social, cloud storage
  • Network: routers, firewalls, servers
  • People: interviews (what to look for)

Output

A list of targets with priorities (what to collect first and why).

07 / Procedure

Preservation (3/9)

Protect evidence from change, damage, or contamination.

Do

  • Photograph and label evidence
  • Use write blockers for storage devices
  • Seal and store items securely
  • Record every transfer (chain of custody)

Don’t

  • Boot and “click around” on the original device
  • Install tools on the suspect system
  • Share evidence copies without permission
07 / Procedure

Collection (4/9)

Collect devices and data sources safely, keeping good notes.

What to record

  • Device identifiers (serial, model), owner, location found
  • Power state (on/off), screen status, network connections
  • Who collected it and when
  • Packaging and transport details

Tip

If a device is powered on, decide quickly whether to capture volatile data first.

07 / Procedure

Acquisition / Imaging (5/9)

Create forensic copies so analysis does not change the original.

Disk imaging

  • Use write blockers
  • Bitstream (sector-by-sector) image when required
  • Hash before/after to prove integrity

Mobile acquisition

  • Logical vs physical (depends on device/security)
  • Keep device state stable (airplane mode if allowed)
  • Document tools and settings used
07 / Procedure

Examination (6/9)

Extract and organize data for analysis (without jumping to conclusions).

Tasks

  • File listing, keyword search, timeline building
  • Recover deleted files (where possible)
  • Parse artifacts (browser, chat DBs, logs)
  • Normalize timezones and formats

Output

A structured set of findings and artifacts ready for analysis.

07 / Procedure

Analysis (7/9)

Answer the case questions by connecting artifacts into a clear story supported by evidence.

Good analysis

  • Uses multiple artifacts to confirm events
  • Separates facts vs. assumptions
  • Includes alternative explanations when relevant

Common mistakes

  • Relying on a single artifact
  • Ignoring timezone differences
  • Overstating what the evidence proves
07 / Procedure

Reporting (8/9)

Write a report that another expert can reproduce and a judge/jury can understand.

Include

  • Scope and authority
  • Tools + versions + settings
  • Hashes and chain-of-custody summary
  • Findings with supporting artifacts

Style

Clear, neutral, and factual. Avoid guessing.

07 / Procedure

Presentation & Archiving (9/9)

Present results when needed, then store evidence and documentation securely.

Presentation

  • Explain method and integrity (hashes)
  • Use simple visuals: timelines, diagrams
  • Be prepared to defend your process

Archiving

  • Store originals and images securely
  • Maintain chain of custody until closure
  • Follow retention policy and law

09 / Network Forensics

Network Forensics: What It Is

Network forensics uses traffic and logs to reconstruct what happened across systems.

Questions it answers

  • Who communicated? (IPs, hostnames)
  • When? (timestamps, sessions)
  • How? (protocols, ports)
  • Sometimes what? (payloads — depends on encryption)

Limitations

  • Encryption hides content (HTTPS, VPN).
  • NAT can hide internal hosts.
  • Missing logs = missing visibility.
  • Packets show activity, not intent — you still need context.
09 / Network Forensics

Network Evidence: Where It Comes From

Different sources give different detail. You usually combine them.

Common sources

PCAP (raw packets)
NetFlow (session summaries)
Firewall / IDS alerts
DNS logs / proxy logs
Server access logs
Endpoint telemetry (EDR)

Collection tips

  • Record timezone and clock accuracy.
  • Keep originals read-only; work from copies.
  • Hash exported log/pcap files and track chain of custody.
  • Note interfaces/ports and capture filters used.
10 / PCAP Analysis

PCAP: What It Is (1/2)

A PCAP stores captured packets, including headers and sometimes payloads.

What’s inside a packet?

  • Link + IP + transport headers (Ethernet/IP/TCP/UDP)
  • Ports (80, 443, 53, …)
  • Protocol data (DNS queries, HTTP requests, TLS handshakes)

How it’s collected (high level)

  • On an endpoint (capture on a NIC)
  • On a switch (SPAN / mirror port)
  • On a tap/sensor (network probe)
  • On a server (capture server-side traffic)
10 / PCAP Analysis

PCAP Analysis: How It’s Done (2/2)

A simple workflow students can follow every time.

Step-by-step workflow

  1. Confirm timeframe + timezone.
  2. Find “top talkers”: IPs, ports, protocols.
  3. Filter by suspect host: `ip.addr == x.x.x.x`.
  4. Inspect DNS: queries, responses, unusual domains.
  5. Check HTTP (if not encrypted): requests, files, user agents.
  6. For TCP sessions: “follow stream” to view a conversation.
  7. Write notes: why you think it matters + evidence lines.

Common red flags

  • Many failed logins (brute force)
  • Unexpected outbound connections
  • Large uploads to unknown hosts
  • DNS to random-looking domains
  • Traffic at strange hours

16 / Tools Suite

Forensic Acquisition Tools

Applications used to safely capture, clone, and preserve data with mathematical integrity.

GUI IMAGER

FTK Imager

The industry standard for quick disk imaging, directory mounting, and hash calculation. Known for reliability on Windows environments.

LINUX COMPLIANT

Guymager

A fast, open-source acquisition tool for Linux systems. Supports multi-threaded processing and writes out EWF (E01) files.

TERMINAL ENGINE

dd & dcfldd

Low-level command-line disk dumping utilities. `dcfldd` is customized for forensics, generating progress bars and on-the-fly hashes.
