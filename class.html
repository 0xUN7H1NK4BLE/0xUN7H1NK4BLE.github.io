The File System Hierarchy

STRUCTURE

Windows has C:\. Linux has / (Root). It is a tree structure.

$ ls /
bin dev home lib64 mnt proc run srv tmp var
boot etc lib media opt root sbin sys usr

Critical Directories

STRUCTURE

/bin (Binaries)

Contains the programs you run like ls, cp, cat.

/etc (Configuration)

Settings files. E.g., /etc/passwd (users), /etc/ssh (remote access).

/home (Users)

Personal files for users. E.g., /home/alex.

/var (Variable)

Logs and Websites. E.g., /var/log (Forensics Goldmine!).

/tmp and Malware

SECURITY INSIGHT

The /tmp Danger

The /tmp directory is one of the few places ANY user can write files to.

Forensic Tip: Hackers often hide scripts here because permissions are open.

# Checking for hidden scripts in tmp
$ ls -la /tmp
.X11-unix
.ice-unix
.hidden_miner.sh <-- Suspicious!

User Types

USERS

Root (UID 0)

The Administrator. Has unrestricted access to everything.

System Users

Run background services (e.g., www-data for web servers).

Regular Users

Human users (UID 1000+). Have restricted access.

User Management Commands

USERS
# 1. Create a user (with home folder)
$ sudo useradd -m alex

# 2. Set password
$ sudo passwd alex

# 3. Switch to that user
$ su - alex

# 4. Delete user
$ sudo userdel -r alex

Permissions: rwx

PERMISSIONS

Every file has 3 sets of permissions: User, Group, Others.

Read (r)

View file contents.

Value: 4

Write (w)

Modify/Delete file.

Value: 2

Execute (x)

Run file as program.

Value: 1

Octal Math

PERMISSIONS

Calculation

rwx = 4+2+1 = 7

rw- = 4+2+0 = 6

r-x = 4+0+1 = 5

r-- = 4+0+0 = 4

# Common Codes
777 : Everyone Full Access (Danger)
755 : Owner Full, Others Read/Run (Standard)
700 : Owner Only (Private)

Changing Permissions: chmod

COMMANDS
# Numeric Mode
$ chmod 700 secret.txt
# Symbolic Mode (User + Execute)
$ chmod u+x script.sh
# Symbolic Mode (Group - Write)
$ chmod g-w file.txt

Changing Ownership: chown

COMMANDS

Give a file to another user.

# Syntax: chown user:group file

$ sudo chown alex report.pdf

# Recursive (entire folder)
$ sudo chown -R alex:developers project_folder/

Secure the File

  2. Check permissions: ls -l secret.txt
  3. Lock it down so ONLY you can read/write it: chmod 600 secret.txt

  10. Create a file shared.txt and give read permission to group and others: chmod 644 shared.txt
  11. Check the octal value: What does chmod 644 mean? (r--r--r--)
  12. Remove write permission from group on shared.txt using symbolic mode: chmod g-w shared.txt
  13. Add execute permission for owner using symbolic mode: chmod u+x shared.txt
  14. Create a file test.txt and change its owner (requires sudo): sudo chown root test.txt
  15. Verify ownership change: ls -l test.txt (should show root as owner)
  16. Change ownership back to your user: sudo chown $USER test.txt
# Example: Understanding permission output
$ ls -l secret.txt
-rw------- 1 kali kali 0 Jan 10 10:00 secret.txt
# rw------- = 600 (owner: read+write, group: none, others: none)

Security Note: Always use the minimum permissions necessary. Files with 777 (rwxrwxrwx) are a security risk!

IAM Quiz

REVIEW
1. What Octal value is Read + Execute?
5 (4 + 1)
2. Which command changes file ownership?
chown
3. What does the AAA model stand for?
Authentication, Authorization, and Accounting
Authentication = Who are you? Authorization = What can you do? Accounting = What did you do?
4. What is the UID (User ID) of the root user?
0
Root always has UID 0, giving it unrestricted access to the system.
5. What does chmod 755 mean in terms of permissions?
Owner: rwx (7), Group: r-x (5), Others: r-x (5)
Owner has full access, group and others can read and execute but not write.
6. Which command creates a new user with a home directory?
sudo useradd -m username
The -m flag creates the home directory.
7. What is the difference between su and sudo?
su switches to another user (requires that user's password), while sudo runs a single command as another user (requires your password and sudo privileges).
8. What permission value represents "Read + Write"?
6 (4 + 2)
Read (4) + Write (2) = 6
9. What does chmod 700 do?
Owner: rwx (full access), Group: --- (no access), Others: --- (no access)
Only the owner can read, write, or execute the file.
10. How do you change file ownership and group at the same time?
sudo chown user:group filename
Example: sudo chown alex:developers file.txt
11. What is the symbolic mode equivalent of chmod 644?
chmod u=rw,go=r filename or chmod u+rw,go+r filename
Owner: read+write, Group: read, Others: read
12. What type of users typically have UID 1000+?
Regular users
System users usually have UIDs below 1000, while regular human users start at 1000.
13. What does the "x" permission mean for a directory?
Execute permission on a directory means you can "enter" or "cd" into it
Without execute permission, you cannot access the directory even if you have read permission.
14. Which command deletes a user and their home directory?
sudo userdel -r username
The -r flag removes the user's home directory and mail spool.

Hashing (Digital Fingerprint)

INTEGRITY

One-way math. Cannot reverse a hash to get the data.

$ echo "Hello" > msg.txt
$ md5sum msg.txt
b1946ac92492d2347c6235b4d2611184 msg.txt

$ echo "Hello!" > msg.txt
$ md5sum msg.txt
5a8dd3ad0756a93ded72b823b19dd877 msg.txt

Changing 1 character changes the entire hash.

Salting Passwords

INTEGRITY

The Problem

If two users have the password "password123", they have the same hash. Hackers use "Rainbow Tables" to reverse these common hashes.

The Solution: Salt

Add random data to the password before hashing.

Hash("password123" + "Xy9z")

Now every hash is unique.

Task: Hashing & Integrity Check

PRACTICAL TASK

Verify File Integrity

Practice hashing to detect file tampering:

  1. Create a file evidence.txt with the content "This is critical evidence".
  2. Generate MD5 hash: md5sum evidence.txt > evidence.md5
  3. Generate SHA256 hash: sha256sum evidence.txt > evidence.sha256
  4. View both hash files to see the difference in hash algorithms.
  5. Modify evidence.txt slightly (add a space or change one letter).
  6. Run md5sum -c evidence.md5 to verify - it should FAIL!
  7. Generate new hashes and compare with the originals.

Question: Why did the hash change completely even though you only changed one character?

Encoding vs Encryption

CONCEPTS

Encoding (Base64)

NOT Security. Just a format change.

$ echo "Admin" | base64
QWRtaW4K

Encryption (AES)

SECURE. Requires a key/password.

Output looks like garbage binary.

Symmetric Encryption

CONFIDENTIALITY

Shared Key. Same password to Lock and Unlock.

# Encrypt
$ openssl aes-256-cbc -in data.txt -out data.enc
enter aes-256-cbc encryption password:

# Decrypt
$ openssl aes-256-cbc -d -in data.enc -out decrypted.txt

Task: Encoding & Decoding Practice

PRACTICAL TASK

Base64 Encoding

Practice encoding and decoding with Base64. Complete these tasks:

  1. Create a text file with a secret message: echo "My Secret Password: admin123" > secret.txt
  2. Encode the file using Base64: base64 secret.txt > secret_encoded.txt
  3. View the encoded content: cat secret_encoded.txt
  4. Decode it back: base64 -d secret_encoded.txt
  5. Encode a string directly: echo -n "Hello World" | base64
  6. Decode a Base64 string: echo "SGVsbG8gV29ybGQ=" | base64 -d
  7. Compare: Encode "password" and "Password" - notice they produce different outputs.
  8. Create a file with multiple lines and encode it: cat secret.txt | base64
  9. Decode and save to a new file: base64 -d secret_encoded.txt > decoded_secret.txt
  10. Verify the original and decoded files match: diff secret.txt decoded_secret.txt
# Example: Quick encode/decode
$ echo "test" | base64
dGVzdAo=
$ echo "dGVzdAo=" | base64 -d
test

Important: Base64 is NOT encryption! It's just encoding. Anyone can decode it easily. Never use Base64 for security!

Task: Encrypt & Decrypt Files

PRACTICAL TASK

Practice Encryption

Secure sensitive data using symmetric encryption:

  1. Create a file sensitive_data.txt with some confidential information.
  2. Encrypt it using AES-256-CBC: openssl aes-256-cbc -in sensitive_data.txt -out sensitive_data.enc
  3. Enter a strong password when prompted (remember it!).
  4. Verify the encrypted file exists: ls -lh sensitive_data.enc
  5. Try to read the encrypted file: cat sensitive_data.enc (should be unreadable).
  6. Decrypt the file: openssl aes-256-cbc -d -in sensitive_data.enc -out decrypted_data.txt
  7. Compare the original and decrypted files: diff sensitive_data.txt decrypted_data.txt
  8. Delete the original unencrypted file: rm sensitive_data.txt

Security Note: In real scenarios, always securely delete original files after encryption!

Asymmetric Encryption (PKI)

CONFIDENTIALITY

Public Key

Share with everyone.

Used to ENCRYPT messages sent to you.

Private Key

Keep Secret. Never Share.

Used to DECRYPT messages.

Digital Signatures

AUTHENTICITY

How do I know the message truly came from you?

Signing

Sender encrypts a hash of the file using their PRIVATE KEY.

Verifying

Receiver decrypts it using the Sender's PUBLIC KEY.

If it decrypts correctly, it MUST have come from the owner of the Private Key.

PKI & Certificates

TRUST

Who do we trust? The Chain of Trust.

Root CA

Certificate Authority (e.g., DigiCert). The ultimate source of trust.

Intermediate CA

Delegated trust.

End Entity

The website (e.g., google.com). Has a certificate signed by the CA.

Crypto Attacks

ATTACKS

Brute Force

Try every possible combination (aaaa, aaab...). Slow.

Dictionary Attack

Try words from a list (password, 123456). Fast.

Rainbow Table

Pre-computed list of hashes. Instant, if no salt is used.

Steganography

HIDING DATA

Security through Obscurity. Hiding text inside images.

# Embed
$ steghide embed -cf image.jpg -ef secret.txt

# Extract
$ steghide extract -sf image.jpg

Risk Management

RISK

Calculation: Risk = Likelihood x Impact

High Risk

High Impact + High Likelihood.

Example: Public S3 bucket with passwords.

Medium Risk

High Impact + Low Likelihood.

Example: Earthquake destroying data center.

Low Risk

Low Impact + Low Likelihood.

Incident Response (NIST)

RESPONSE

What do we do when we are hacked?

  1. Preparation: Plan and Tools ready.
  2. Detection & Analysis: Find the breach.
  3. Containment, Eradication & Recovery: Stop it, remove it, restore backups.
  4. Post-Incident Activity: Lessons learned.

Task: Incident Response Simulation

PRACTICAL TASK

Practice Incident Response

Simulate an incident response scenario:

  1. Detection: Check recent authentication attempts: tail -n 50 /var/log/auth.log | grep "Failed"
  2. Analysis: Find all failed login attempts and count them: grep "Failed password" /var/log/auth.log | wc -l
  3. Containment: Check current network connections: netstat -tulpn or ss -tulpn
  4. Evidence Collection: Create an evidence folder and copy suspicious log entries: mkdir -p ~/incident_evidence
  5. Save suspicious log entries: grep "Failed password" /var/log/auth.log > ~/incident_evidence/failed_logins.txt
  6. Documentation: Create a report file with timestamp: echo "Incident Report - $(date)" > ~/incident_evidence/report.txt
  7. Add findings to the report: echo "Failed login attempts found: $(grep -c 'Failed password' /var/log/auth.log)" >> ~/incident_evidence/report.txt
  8. Preservation: Create a hash of all evidence files: sha256sum ~/incident_evidence/* > ~/incident_evidence/evidence_hashes.txt

Learning: This demonstrates the NIST incident response framework in action!
